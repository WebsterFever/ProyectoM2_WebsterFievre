# API MiniBlog — Development Roadmap

This README is the **step-by-step guide** for building the project, not the finished
solution. The idea is that each step gets implemented, tested, and understood before
moving on to the next one. We'll go through it in **9 steps**.

Stack: Node.js + Express + PostgreSQL (`pg`) + Supertest + OpenAPI + deploy on Railway.

---

## Local configuration (`.env`)

The project reads the database configuration from environment variables (never
hardcoded). Copy `.env.example` to `.env` and fill it in with your local Postgres
values:

```bash
cp .env.example .env
```

| Variable | What it's for | Example |
|---|---|---|
| `PORT` | Port the Express server runs on | `3000` |
| `DB_HOST` | Postgres host | `localhost` |
| `DB_PORT` | Postgres port | `5432` |
| `DB_NAME` | Database name | `miniblogapi` |
| `DB_USER` | Postgres user | `postgres` |
| `DB_PASSWORD` | Postgres password | `your_local_password` |
| `DB_MAX_CONNECT` | Max simultaneous connections in the pool (`pg.Pool`) | `20` |
| `DB_IDLETIMEOUT` | Time (ms) an idle connection stays open before closing | `30000` |
| `DB_CONNECTIONTIMEOUT` | Time (ms) the pool waits for a free connection before giving up | `2000` |

`.env` is **never** committed (it's in `.gitignore`) — only `.env.example`, with empty
keys, serves as a reference for whoever runs the project.

## How to run locally

1. Install dependencies:
   ```bash
   npm install
   ```
2. Configure `.env` (see section above).
3. Start the server:
   ```bash
   npm run dev    # with auto-reload (nodemon)
   # or
   npm start      # without reload
   ```
   On first run, the server confirms the connection to Postgres and automatically
   creates the `authors`/`posts` tables if they don't already exist
   (`src/config/initDb.js`). `sql/setup.sql` does the same thing and serves as a
   reference in case you prefer to run it manually.
4. The API comes up at `http://localhost:3000`.

## How to run tests

```bash
npm test
```

Runs the suite with Vitest, which includes:
- **Unit tests** for the validation middlewares (`tests/validate*.test.js`) — they
  don't depend on the database or the HTTP server.
- **Integration tests** with Supertest (`tests/*.integration.test.js`) — they spin up
  the real Express app and use the Postgres configured in `.env`. They clean up the
  data they create at the end (`afterAll`) and use unique values (e.g. an email with a
  timestamp), so running `npm test` several times in a row doesn't break due to
  duplicate data.

## OpenAPI documentation

With the server running, go to **`http://localhost:3000/docs`** for interactive
documentation (Swagger UI), generated from the spec defined in `src/config/swagger.js`
(the `swaggerSpec` object, in OpenAPI 3.0 format).

## Deploy on Railway

1. Create an account at [railway.app](https://railway.app) and a new project.
2. Add a **PostgreSQL** service to the project — Railway automatically provisions the
   database and exposes its own connection variables.
3. Add a second service from the miniBlogAPI GitHub repository (deploy via Git).
4. On the API service, configure the environment variables `DB_HOST`, `DB_PORT`,
   `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_MAX_CONNECT`, `DB_IDLETIMEOUT`, and
   `DB_CONNECTIONTIMEOUT`:
   - Use the **internal URL**/variables from Railway's Postgres service (it provides
     `PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD` — reference them or copy
     the values into the variables the project expects).
   - Don't set `PORT` manually: Railway injects its own port and `index.js` already
     uses `process.env.PORT`.
5. Railway builds and runs `npm start` automatically based on `package.json`.
6. After deploying, grab the **public URL** generated by Railway (Settings →
   Networking) and test the endpoints, e.g.: `https://your-project.up.railway.app/authors`
   and `https://your-project.up.railway.app/docs`.

## Step 1 — Understand the domain before writing any code

Before touching the keyboard, answer in writing (it can be right here, in your own
notes):

- What are the entities? (`authors`, `posts` — and later `comments` in the extra
  credit)
- What's the relationship between them? (1 author → N posts — a 1:N relationship via
  `author_id` as FK)
- What happens to the posts of an author who gets deleted? (the sample script uses
  `ON DELETE CASCADE` — understand what that means before copying it)
- Which fields are required on each entity, and which have special rules (e.g. a
  unique `email`)?

**Why**: if you can't answer this off the top of your head, you'll end up copying SQL
and code without knowing why it's shaped that way, and you'll get stuck on the first
requirement change.

**Criteria to move on**: you can sketch both tables with their columns and the FK
arrow on paper/diagram without looking at the assignment.

## Step 2 — Project folder structure

Define (and create empty, just with `.gitkeep` files or a comment) the structure:

```
miniBlogAPI/
├── src/
│   ├── db/            # connection to Postgres (connection pool)
│   ├── routes/        # defines the endpoints (the "what" of the URL)
│   ├── controllers/   # reads req/res, calls the service, decides the status code
│   ├── services/       # business logic + SQL queries
│   ├── middlewares/    # error handling, validations
│   └── app.js         # sets up Express and the middlewares
├── sql/
│   ├── setup.sql      # CREATE TABLE
│   └── seed.sql        # sample data INSERTs
├── tests/
├── index.js            # just starts the server (requires app.js)
├── .env.example
└── README.md
```

**Why separate `routes` from `controllers` from `services`**: the route only decides
"which URL/verb triggers what" and delegates to the controller; the **controller**
handles HTTP (`req`, `res`, status code) and calls the service; the **service** doesn't
know anything about HTTP — just business logic and SQL queries. This makes it easier
to test the logic (the service) without needing to simulate HTTP requests.

**Criteria to move on**: the folders exist and you can explain each one's role in one
sentence.

---

## Step 3 — SQL setup and seed scripts, tested directly against Postgres

Write `sql/setup.sql` (CREATE TABLE for `authors` and `posts`, with PK, FK, NOT NULL,
UNIQUE) and `sql/seed.sql` (a few INSERTs).

Then **run this directly in a Postgres client** (psql, DBeaver, TablePlus, whatever you
have) — still without Express in the picture. Manually confirm:

- Both tables were created with the right types.
- An INSERT with a duplicate `email` fails (proves the UNIQUE constraint works).
- An INSERT of a post with a nonexistent `author_id` fails (proves the FK works).
- A DELETE of an author with associated posts behaves the way you expect (cascade or
  error, depending on what you chose).

**Why do this before Express**: if the database modeling is wrong, all the code built
on top of it will inherit the error. It's much cheaper to find that out running raw
SQL than debugging through an API.

**Criteria to move on**: you ran the 4 manual tests above and can explain the result of
each one.

---

## Step 4 — Endpoints with in-memory data (arrays), no database yet

Implement the `authors` and `posts` routes using JS arrays (as in the guide's
example), with no database connection at all. Already set up the full
`routes → controllers → services` flow, even if the "service" is still just an
in-memory array: the route calls the controller, the controller calls the service and
returns the response. Goal: validate the "shape" of the API — HTTP verbs, status
codes, response JSON format — in isolation from any database issue.

Minimum endpoints:
```
GET    /authors
GET    /authors/:id
POST   /authors
PUT    /authors/:id
DELETE /authors/:id

GET    /posts
GET    /posts/:id
GET    /posts/author/:authorId
POST   /posts
PUT    /posts/:id
DELETE /posts/:id
```

Test each one manually with Postman/Thunder Client/curl. Check the status codes: `200`
(success with body), `201` (created), `204` (deleted, no body), `404` (resource not
found).

**Why**: separating "the API responds correctly" from "the database is correct" avoids
debugging two problems at once.

**Criteria to move on**: all 11 endpoints respond with the correct status code and
format using the arrays.

---

## Step 5 — Connect Express to PostgreSQL

Create `src/db/pool.js` with a `Pool` from the `pg` package, reading
host/port/user/password/database name from environment variables (never hardcoded).
Bring up the connection and run a simple query (`SELECT NOW()`) just to confirm the
connection works, before touching the routes.

**Why environment variables**: it's what lets you switch from local database → Railway
without changing code, and it's an explicit requirement of the assignment (don't push
credentials to GitHub).

**Criteria to move on**: running `node` standalone (outside Express), you can import
the pool and print the result of a `SELECT NOW()`.

---

## Step 6 — Swap the arrays for real SQL queries

Now, one endpoint at a time, replace the array with the corresponding query in the
`service` (the `controller` doesn't change at all here — it keeps just calling the
service, without knowing whether there's an array or Postgres behind it). Always use
**parameterized placeholders** (`$1`, `$2`, ...) — never string concatenation in
queries, that's what prevents SQL injection.

Suggested order (from simplest to most dependent):
1. `GET /authors` and `GET /authors/:id`
2. `POST /authors` (here handle the duplicate email error — the database's UNIQUE
   constraint will throw a specific error, catch it and return 400, not 500)
3. `PUT` and `DELETE` for authors
4. Repeat the sequence for `posts`, including `GET /posts/author/:authorId` (which
   needs a `WHERE author_id = $1`)

After migrating each endpoint, test it again in Postman before moving to the next one
— don't migrate all of them at once and only then test, because then it's hard to tell
which query broke.

**Criteria to move on**: all 11 endpoints work against the real database, with the
same status codes validated in Step 4.

---

## Step 7 — Validations and a global error middleware

Add validations before hitting the database (required fields not empty, correct
types) and a centralized error middleware in Express (the one with 4 arguments:
`(err, req, res, next)`) that translates exceptions into JSON responses with the right
status code (`400`, `404`, `500`).

**Why centralize it**: it avoids repeated `try/catch` in every route and guarantees the
API never "forgets" to respond something in case of an error (a common mistake
mentioned in the assignment: "always responding 200 even on error").

**Criteria to move on**: sending a POST without `name` returns 400 with a clear
message; looking up a nonexistent `:id` returns 404; forcing a database error doesn't
crash the server or return 200.

---

## Step 8 — Automated tests with Supertest

Write at least 6 tests covering, at minimum:
- Create author (success)
- Look up an existing author
- Create post (success)
- Delete a nonexistent resource (expects 404, not 200)
- Create author with duplicate email (expects 400)
- Create post without a required field (expects 400)

Set up the `npm test` script to run everything via Supertest + Jest (or whichever
runner you prefer).

**Why test error cases, not just the happy path**: that's where most bugs in
poorly-written APIs show up — returning 200 when it should be 404/400.

**Criteria to move on**: `npm test` runs and all tests pass reproducibly (running it
twice in a row shouldn't break because of duplicate data from the previous test —
think about how to clean up state between runs).

---

## Step 9 — Documentation, `.env.example`, and deploy on Railway

- Document the endpoints in OpenAPI 3.0 format (spec defined as a JS object in
  `src/config/swagger.js`, served via Swagger UI at `/docs`).
- Create `.env.example` with the keys (no real values) the application expects.
- Update the main `README.md` with: how to run locally, how to run tests, how to view
  the OpenAPI docs, and the step-by-step deploy on Railway (environment variables,
  internal URL, public URL).
- Confirm `.env` is in `.gitignore` before the first commit with real credentials
  nearby.
- Deploy the project on Railway, plugging in their Postgres and the same env vars used
  locally.

**Criteria for completion**: someone outside the project, following only your
README, can clone the repo, run it locally, and run the tests without asking you
anything.

## Purpose of this guide

This guide aims to walk you through the technical implementation of the Integrative
Project's backend.

It doesn't replace the official assignment or define a single way to solve the
project. Its purpose is to guide you with concrete examples of structure, SQL
scripts, and initial data you can use as a reference to organize your development.

The focus is on helping you build a functional, well-structured REST API correctly
connected to PostgreSQL.

## What's expected of the project

You're expected to develop a REST API in Node.js + Express that:

- Implements CRUD operations for the `authors` and `posts` entities.
- Persists data in PostgreSQL.
- Handles basic validations and errors.
- Includes minimal documentation and automated tests.

Beyond the number of features, special value will be placed on:

- The endpoints working correctly.
- The database connection being properly configured.
- SQL queries being parameterized.
- The project being properly versioned.
- The documentation allowing the project to be run without external help.

## How to approach the development

**Before writing code:**

1. Read the full assignment.
2. Identify the entities and relationships (`authors` → `posts`).
3. Define a simple folder structure (`routes`, `services`, `db`, `middlewares`).
4. Decide which endpoints you'll implement first.
5. Prepare your database script before connecting Express.

**During development**, it's recommended to proceed in this logical order:

1. Create the SQL setup and seed script.
2. Test the tables directly in PostgreSQL.
3. Create the endpoints with in-memory data (arrays).
4. Connect the application to PostgreSQL using `pg`.
5. Replace the arrays with real database queries.
6. Add validations and error-handling middleware.
7. Add tests.
8. Document with OpenAPI.
9. Prepare the deployment.

Test the endpoints frequently with tools like Postman or Thunder Client. Make small,
descriptive commits.

## Example setup and seed script (.sql)

You can use the following example as a base for your database initialization file.

```sql
CREATE TABLE authors (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(150) UNIQUE NOT NULL,
  bio TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE posts (
  id SERIAL PRIMARY KEY,
  title VARCHAR(200) NOT NULL,
  content TEXT NOT NULL,
  author_id INTEGER NOT NULL,
  published BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  FOREIGN KEY (author_id) REFERENCES authors(id) ON DELETE CASCADE
);
```

**Sample data (seed):**

```sql
INSERT INTO authors (name, email, bio) VALUES

  ('Carlos Ruiz', 'carlos@example.com', 'Technical writer specialized in databases'),
```

Before connecting your application, verify that:

- The tables were created correctly.
- The foreign keys work.
- You can run CRUD queries manually.

## Recommended best practices

**Architecture:**
- Separate routes (`routes`) from business logic (`services`).
- Centralize the database connection.
- Implement a global error-handling middleware.

**SQL:**
- Use parameterized queries (`$1`, `$2`, etc.).
- Don't concatenate strings to build queries.
- Properly handle cases where there are no results (404).

**Validations:**
- Check required fields.
- Make sure the email is unique.
- Return appropriate HTTP status codes (400, 404, 201, etc.).

**Tests:**
- Cover at least: create author, get author, create post, delete nonexistent
  resource.
- Use supertest to test HTTP endpoints.

**Git:**
- Don't push your `.env` file.
- Include a `.env.example`.
- Keep the repository public.

## Common mistakes to avoid

- Connecting Express to PostgreSQL without having tested the SQL script first.
- Not handling database errors.
- Not validating inputs before making the query.
- Always responding 200 even when there are errors.
- Not documenting how to run the project.
